# JAPAN EXPORT 事業モデル刷新仕様書 v1.0

## 背景と目的

現在の JAPAN EXPORT 事業は、ジャンル横断で単一ロジックが動く構造となっており、2026年4月に発生した複数のインシデント（全約824件の出品が Out of Stock 化、誤判定による二重販売、API rate limit 枯渇）に示されたとおり、曖昧な状態での自動処理が致命的なリスクを生んでいる。

本仕様書は、事業を「学習する組織モデル」へ刷新するための設計書である。目的は以下の3点に集約される。

1. リサーチと出品の精度を構造的に向上させる
2. 異常検知時に安全側へフォールバックする全自動運用を確立する
3. 時間経過とともに全体が賢くなる学習ループを内蔵する

本仕様書は Claude Code による実装の指示書としても、Daichi をはじめとする関係者との仕様共有資料としても使用される。

---

## 全体アーキテクチャ

本システムは6層構造を持つ。上位から下位へ順に以下のとおり。

### Capital Allocation 層（経営層）

各現場部署の ROI、回転日数、Sell-through 率を週次で評価し、翌週の仕入予算を動的に配分する。成績優秀な部署に資金を厚く、停滞部署は絞る。配分変動は週次上限20%までとし、短期成績による資金集中の暴走を防ぐ。

### 共通ルール層（管理部）

全部署共通の基盤ルールと計算式のみを保持する。個別パラメータ（送料、重量など）は持たない。

保持するもの:

- タイトル規則（80字上限、SHIPPING WORLDWIDE 必須記載）
- 為替レート（155 JPY/USD）
- 手数料率（FVF 13.25%、国際 1.35%、Payoneer 2%、Promoted 3%、合計19.6%）
- 利益計算式: 売価 × 0.804 − 仕入原価 − 送料
- 価格上限（1商品あたりの最大出品価格）
- カテゴリ混在禁止ルール（ワンピースをポケモンカテゴリに、またその逆も禁止）
- 全社ガードレール指標（キャンセル率、ネガティブ率、API 失敗率、出品エラー率）

### 現場部署層（Layer 3）

シリーズ単位で1部署を構成する実行単位。cron と AI agent のペアで稼働する。

配置例:

- One Piece Card 部
- カードダス部
- ポケモン部
- BBM 野球部
- フィギュア部（新カテゴリ例）
- 時計部（新カテゴリ例）

各部署は以下のパラメータを個別に保持する:

- 送料（例: カード系 ¥3,000、フィギュア ¥5,000、時計 ¥2,500）
- 想定重量（g）
- 発送方法（FedEx 小型 / 中型 / EMS 等）
- 破損リスク係数
- 5軸スコアの重み付け（部署特性に応じて調整）

各部署は以下の工程を一気通貫で担当する:

1. Mercari 巡回（cron）
2. 候補商品の5軸スコア判定（AI agent）
3. eBay 出品（cron）
4. 監視と異常検知（cron + AI agent）
5. 異常時の自動フォールバック

### 製品タグ層（商品データベース）

キャラクター、状態（PSA10 など）、季節、バージョンを商品メタデータとして横断管理する。部署をまたいで商品を検索・分析できるようにする。

Layer 4（キャラクター）および Layer 5（状態）は部署化せず、このタグ層にデータとして保持する。特定タグが粗利条件を満たした場合のみ、部署内にタグ専属サブロジックを追加する方式とする。

### Intelligence 層（分析部）

全部署の稼働ログ、売買データ、失敗理由タグを一元集約し、AI が横断パターンを抽出する。

発見された知見は共通ルール層へ「昇格提案」として送られ、ユーザー承認の上で全部署に即時反映される。

過学習防止のため、昇格には最低3部署での再現性を必須条件とする。

### Market Signal 層（市場調査部）

外部イベントを監視し、該当部署へ事前アラートを発する。

監視対象:

- アニメ新シーズン放送スケジュール
- 大会（TCG 公式大会）の日程と優勝者
- 為替変動（閾値超えでアラート）
- 他プラットフォーム（Yahoo オークション、駿河屋）の相場推移

誤検知フィルタ: 外部シグナル単独では仕入トリガーにせず、部署側の内部データと整合した場合にのみ発火する。

## 3つの循環

本システムは以下の3つのデータ循環で駆動される。

**データ循環**: 現場部署の稼働結果 → 製品タグ層で集計 → Intelligence 層で分析 → 共通ルール層へ昇格 → 全部署へ反映

**資本循環**: 部署別 ROI・回転日数 → Capital Allocation 層が評価 → 翌週の仕入予算を再配分 → 部署稼働

**外部循環**: Market Signal 層が先行シグナル取得 → 該当部署へアラート → 仕入強化・価格修正を事前実行

## リサーチエンジン仕様

### 5軸スコアリング

Mercari 巡回で拾った候補商品は、以下の5軸で0〜100点に評価される。部署ごとの重みで加重合計し、閾値を超えた場合のみ仕入れを実行する。

| 軸 | 内容 | データ源 |
|---|---|---|
| 相場乖離 | 直近30日平均売価との差分 | eBay Sold |
| 回転速度 | 類似商品の平均 Sell-through 日数 | eBay Sold |
| 競合密度 | 同一商品の現行出品数 | eBay Active |
| 需要トレンド | 30日/90日の価格推移方向 | eBay Sold 時系列 |
| 季節・イベント | 発売記念日、アニメ放送、大会日程 | Market Signal 層 |

### 類似商品クラスタリング

対象商品の売買データが薄い場合、商品ベクトル化によって類似クラスタ内の相場を参照する。これによりデータ不足商品でも精度を保つ。

### 競合セラー分析

個別商品ではなく競合セラー単位で動向を追跡する。以下を自動検知する:

- 大量出品による相場下落リスク
- 撤退による仕入機会
- 値下げ頻度の上昇

### 失敗理由タグ化

出品した全商品に以下のメタデータを紐づけて保存する:

- 出品時の5軸スコア
- 出品文、価格、写真枚数
- 結果: 売れた日数、最終売価、キャンセル有無
- 売れなかった場合の失敗理由（AI による分類: タイトル不備 / 写真品質 / 価格設定ミス / 相場外 / 需要蒸発 / 季節外れ）

Intelligence 層がこれらを横断分析し、スコアの重みや閾値を自動調整する学習ループを形成する。

## 異常対応: 全自動フォールバック

「全自動」とは AI が判断して進めることではなく、異常発生時に事前定義された安全側のデフォルト動作へ自動フォールバックすることを指す。曖昧な状況では proceed せず、hold/停止をデフォルトとする。

| 異常種別 | 検知条件例 | 自動動作 |
|---|---|---|
| 出品エラー | タイトル80字超過、カテゴリ不整合、必須文言欠落、価格上限超過 | 該当出品を自動保留 |
| 需要ズレ | 出品後30日未販売、相場±30%逸脱、過去類似品との大幅乖離 | 複合ガード付き自動価格調整 |
| システム障害 | VPS Playwright タイムアウト、API rate limit、Sheets 同期失敗、在庫判定不一致 | 該当処理を自動停止 |

全異常は非同期ログへ集約される。ユーザーは週次レビューで内容確認し、月次でパラメータ調整を行う。リアルタイムでの介入は行わない。

### 自動価格調整の複合ガード

単純な値下げを避けるため、以下を順に判定する:

1. 原因判別（相場比較 / 競合分析 / 出品文診断）
2. 下げ幅上限（仕入原価の n% までしか下げない）
3. 段階的撤退（N ヶ月未売却で自動削除）
4. 再出品禁止リスト（繰り返し失敗した SKU は除外）
5. 知見昇格（複数部署で同じ失敗が検出されれば共通ルール層へ昇格）

## 新部署判断基準

| 基準 | 条件 |
|---|---|
| Entry（新 Layer 3 部署新設） | 事前リサーチで想定月次粗利 ¥50,000 以上 |
| Expand（部署内にタグ専属ロジック追加） | 特定キャラ/状態で単月粗利 ¥100,000 かつ月次出品30件以上を3ヶ月連続達成 |
| Exit（撤退） | 3ヶ月連続で粗利 ¥20,000 未満、またはキャンセル率5%超 |

Expand 基準を満たすまでは Layer 3 の親部署で扱う＝細分化を遅延させる設計によって、部署乱立を防ぐ。Exit 条件を明記することで、惰性で死んだ部署を残さない。

## PDCA 指標（3層構造）

| 層 | 指標 | 用途 |
|---|---|---|
| Primary（必達） | 月次粗利（¥） | 部署存続の判断 |
| Secondary（質） | 利益率%、Sell-through 率、平均回転日数 | 改善点の特定 |
| Guardrail（事故防止） | キャンセル率、ネガティブ率、出品エラー率、API 失敗率 | 暴走・事故検知 |

サイクル:

- 週次: 数値レビュー
- 月次: 部署ごとの継続・拡大・撤退判断
- 四半期: 全体構成の見直し

Primary 指標のみを見ると、売上が立ちながらアカウントが破壊されるインシデント（4月1日型）が再発する。Guardrail は必ず併設する。

## 導入フェーズ（順序必須）

以下の順序で導入する。前フェーズ完了を次フェーズ着手の条件とする。

**Phase 0: 前提条件の解消（着手条件）**

- Mercari VPS タイムアウト（Page.goto 30000ms exceeded）の解消
- retry + 二段階検証ロジックの実装
- 環境要因で判定が汚染されない設計の確立

この Phase 0 を完了せずに新モデルへ進むことは禁止する。土台が不安定なまま上物を建てると4月1日型のインシデントが再発する。

**Phase 1: 共通ルール層 + ステージング環境構築**

全部署が参照する共通ルール層を実装する。本番投入前にロジックを検証できるステージング環境を最初から組み込む。テスト環境なしの本番投入は禁止する。

**Phase 2: Layer 3 部署を1つ試験運用**

最初に One Piece Card 部を対象として、5軸スコアリング、失敗理由タグ化、全自動フォールバックを含む完全な Layer 3 部署を構築する。

**Phase 3: 検証済み設計を他 Layer 3 部署へ横展開**

Phase 2 で検証された部署テンプレートを、カードダス部、ポケモン部、BBM 野球部などへ順次展開する。

**Phase 4: 新カテゴリへ拡張**

時計、カメラ、カーパーツ、フィギュアなど新カテゴリへ拡張する。新カテゴリごとに部署パラメータ（送料、重量、発送方法、破損リスク係数）を設定する。

## 絶対遵守事項

以下は、いかなる状況でも変更・省略・妥協してはならない。

1. 曖昧な状態（VPS タイムアウト、ネットワークエラー等）で「売れた」「エラー」と断定判定しない。必ず retry + 二段階検証を経る。
2. Phase 0 未完了の状態で Phase 1 以降に着手しない。
3. ステージング環境で検証していないロジックを本番投入しない。
4. ユーザー承認なしに Intelligence 層の昇格提案を共通ルール層へ反映しない。
5. Capital Allocation 層の配分変動を週次20%上限を超えて実行しない。
6. 全自動フォールバックの判定ロジックをユーザー確認なしに変更しない。
7. カテゴリ混在（ワンピース商品をポケモンカテゴリに出品する等）を絶対に行わない。
8. 2026年4月1日、4月6日、4月10日の全インシデントを記憶し続ける。同型の失敗を二度と起こさない。

## 実装着手時の確認事項

Claude Code が本仕様書に基づいて実装を開始する際は、以下を確認すること:

1. 現状の `mercari_checker.py` および VPS cron の稼働状況
2. 既存の全約824件の出品データと売買履歴への読み取りアクセス
3. ステージング環境の構築可否
4. 本番環境への影響を分離するための手段

不明点がある場合、憶測で実装せず、純之介に確認すること。
